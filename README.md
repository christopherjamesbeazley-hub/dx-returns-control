# DX Returns Control Prototype

Assumption-led browser dashboard for a Philips-style DX Returns Manager / Direct Export returns rollout concept. The app uses clearly synthetic CSV data and does not claim access to Philips internal systems, data, process definitions, or credentials.

The original implementation target was React/Vite. In this workspace, npm registry installs repeatedly timed out, so the delivered prototype is dependency-free and runnable with the Node runtime already present on the machine. The same data model and analytics layer can be moved into React/Vite once package installation is available.

## What It Demonstrates

- Role-specific views for Operations Manager, Coordinator, and Leadership.
- CSV upload for refreshing the in-memory dashboard dataset.
- CSV schema validation for required return-export columns.
- CSV template download for users preparing an export.
- Friendly column alias mapping, such as `Return ID` to `return_id` and `Amount EUR` to `value_eur`.
- Search and pagination for working with the generated scale dataset.
- Open returns register with filters for market, partner, status, and SLA state.
- Overall-aging SLA classification from explicit due dates.
- Missing due dates as a data-quality gap, not an invented SLA result.
- Human-controlled overdue worklist for escalation review.
- Downloadable escalation worklist CSV.
- Downloadable weekly stakeholder report.
- Copyable draft escalation messages for human review.
- Top-10% high-value exposure flag among open returns.
- Weekly digest-style stakeholder summary.
- Deterministic Phase 3 AI-assist summaries that keep humans in control and disclose source fields.
- Prompt-injection guardrails for uploaded notes, delay reasons, status text, and item descriptions.
- Recurring delay theme detection for operational pattern review.
- Phase 4 explainable risk scoring, trend snapshots, early-warning signals, and continuous-improvement actions.
- ROI calculator for a simple annual value estimate.
- About and pilot-proposal panels for reviewer context.
- Local CSS visual treatment with no external image dependency.

## Data Model

The MVP source is `src/data/returns.csv`. Required columns are:

- `return_id`
- `return_type`
- `market`
- `partner`
- `logistics_provider`
- `owner`
- `status`
- `created_date`
- `due_date`
- `last_update_date`
- `closed_date`
- `item_description`
- `quantity`
- `value_eur`
- `currency`
- `delay_reason`
- `notes`
- `source_label`

All records in the committed dataset are fictional and labeled as synthetic demo data. The first 25 records are hand-authored seed examples, and 2,000 additional records are generated by `scripts/generate-returns-data.js` for scale testing.

Uploaded CSV files are loaded only in the browser session. They do not overwrite `src/data/returns.csv`.

## Business Rules

- Snapshot date is fixed at `2026-07-08` for reproducible demo metrics.
- Closed and cancelled returns are excluded from open-return SLA calculations.
- Open returns without `due_date` are classified as `missing-due-date`.
- Open returns due before the snapshot date are `overdue`.
- Open returns due within five days are `near-due`.
- High-value exposure is the top 10% of open returns by `value_eur`.
- Phase 2 exports and draft messages are generated from the loaded dataset for human review only.
- Phase 3 AI-assist outputs are deterministic summaries from loaded CSV fields, not external AI API responses.
- AI-assist confidence is limited when due date, notes, or delay reason are missing.
- AI-assist confidence is also limited when uploaded text resembles prompt injection, jailbreak, secret extraction, or tool-misuse instructions.
- Phase 4 forecast output is deterministic prioritization, not an autonomous predictive model.
- Risk scores show their drivers and are intended for control-meeting review.
- Uploaded CSVs are validated against the expected schema before replacing the in-memory dataset.
- Native Excel `.xlsx` files should be exported as CSV before upload in this dependency-free prototype.
